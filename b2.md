1. Phân tích các thuật toán Rate Limiting phổ biến
Thuật toán 1: Fixed Window Counter (Đếm theo cửa sổ cố định)
Nguyên lý hoạt động: Chia thời gian thành các khoảng (window) cố định có độ dài bằng nhau (ví dụ: mỗi window là 1 phút). Mỗi API Key sẽ có một bộ đếm (counter) gắn liền với window hiện tại. Khi có request đến, hệ thống kiểm tra nếu counter chưa vượt quá 100 thì tăng counter lên 1 và cho phép qua. Nếu vượt quá, request bị từ chối. Khi bước sang window mới, counter được reset về 0.

Cấu trúc dữ liệu: Một Hash Map hoặc cặp Key-Value lưu trữ: String (API_Key:Window_Timestamp) -> Integer (Counter).

Độ phức tạp: Thời gian: O(1), Không gian: O(1) trên mỗi API Key.

Ưu điểm: Cực kỳ dễ triển khai, tốn rất ít bộ nhớ, tốc độ tính toán siêu nhanh.

Nhược điểm: Vấn đề bạo phát ở biên (Burst tại cạnh cửa sổ). Nếu 100 request đổ vào ở giây thứ 59 (cuối window 1) và 100 request khác đổ vào ở giây thứ 01 (đầu window 2), hệ thống phải chịu tải 200 request chỉ trong vòng 2 giây, vi phạm cam kết tối đa 100 request/phút.

Thuật toán 2: Token Bucket (Thùng chứa Token)
Nguyên lý hoạt động: Mỗi API Key sở hữu một chiếc "thùng" chứa tối đa N token (ở đây là 100). Định kỳ theo một tốc độ cố định (ví dụ: cứ mỗi 0.6 giây cấp 1 token), hệ thống sẽ bỏ thêm token vào thùng cho đến khi đầy. Khi có request đến, hệ thống kiểm tra xem thùng còn token không. Nếu còn, lấy ra 1 token và cho phép xử lý. Nếu thùng rỗng, trả về lỗi 429.

Cấu trúc dữ liệu: Lưu trữ trạng thái thùng: String (API_Key) -> Object { tokens: Double, last_refreshed_time: Long }.

Độ phức tạp: Thời gian: O(1), Không gian: O(1) trên mỗi API Key.

Ưu điểm: * Hỗ trợ xử lý bạo phát lưu lượng tốt (Traffic Bursting): Nếu một API Key lâu không dùng, thùng đầy 100 token, họ có thể gửi liên tiếp 100 request ngay lập tức một cách hợp lệ.

Tốc độ cấp phát mượt mà theo thời gian.

Nhược điểm: Trong môi trường phân tán (Distributed Gateway), việc tính toán lại số lượng token dựa trên thời gian thực (last_refreshed_time) có thể gặp vấn đề tranh chấp dữ liệu (Race Condition) khi nhiều instance Gateway cùng cập nhật một key, đòi hỏi cơ chế lock phức tạp hoặc phải sử dụng Lua Script trên Redis.

Thuật toán 3 (Bổ sung): Sliding Window Counter (Đếm theo cửa sổ trượt)
Nguyên lý hoạt động: Đây là sự kết hợp giữa Fixed Window và Sliding Window Log để khắc phục điểm yếu bạo phát ở biên mà không tốn quá nhiều bộ nhớ. Thay vì reset cứng, thuật toán này ước tính số lượng request trong 1 phút gần nhất bằng cách dựa vào counter của window hiện tại và window liền trước.

Lý do lựa chọn:
Chống nghẽn hiệu quả hơn Fixed Window: SafePay không thể mạo hiểm để một đối tác vô tình hoặc cố ý gửi 200 request trong 2 giây ở ranh giới phút cũ và phút mới, vì điều này có thể làm quá tải các service Core Banking phía sau. Sliding Window Counter giải quyết triệt để vấn đề này.

Hiệu năng vượt trội so với Token Bucket phân tán: Để chạy Token Bucket trên cụm distributed (nhiều instance Spring Boot cùng kết nối Redis), ta phải viết Lua Script để đảm bảo tính nguyên tố (Atomicity) khi thực hiện chuỗi thao tác: Lấy thời gian hiện tại -> Tính toán lượng token mới -> Trừ token -> Lưu lại. Việc này tiêu tốn nhiều CPU của Redis. Trong khi đó, Sliding Window Counter chỉ cần các lệnh đơn giản như INCR và EXPIRE của Redis, giúp Gateway đạt throughput lớn hơn đáng kể.

Dễ dàng tối ưu bộ nhớ: Chỉ cần lưu trữ các bộ đếm dạng số nguyên kèm theo TTL (Time-To-Live) tự động xóa của Redis. Khi một API Key không có traffic, nó không chiếm bất kỳ một byte dữ liệu nào trong RAM của hệ thống.

1. Tại sao Local Memory Rate Limiter bị lỗi trong Distributed System?
Khi chạy trên 3 máy chủ (Server A, B, C) độc lập và sử dụng bộ nhớ RAM cục bộ (ví dụ: ConcurrentHashMap hoặc Caffeine Cache), mỗi máy chủ chỉ biết trạng thái của riêng nó mà hoàn toàn không có thông tin về lượng traffic đã đổ vào các máy chủ khác.

Ví dụ thực tế với 120 request:
Giả sử một đối tác (User X) có hạn mức là 100 request/phút. Họ gửi liên tiếp 120 request trong vòng 1 phút, và Load Balancer (ví dụ: Nginx hoặc AWS ALB) áp dụng thuật toán Round-Robin để chia đều lượng traffic này qua 3 server:

Server A nhận: 40 requests → Bộ đếm cục bộ của Server A = 40 (Hợp lệ, vì 40<100).

Server B nhận: 40 requests → Bộ đếm cục bộ của Server B = 40 (Hợp lệ, vì 40<100).

Server C nhận: 40 requests → Bộ đếm cục bộ của Server C = 40 (Hợp lệ, vì 40<100).

Hậu quả: Toàn bộ 120 request đều được xử lý thành công. Hệ thống SafePay đã bị rò rỉ (leak) thêm 20 request vượt ngưỡng cho phép mà không hề hay biết. Trên thực tế, nếu cấu hình Load Balancer chia đều lý tưởng, một User có thể gian lận để gửi tối đa tới 3×100=300 request/ph

Giải thích thiết kế trên Redis:
Redis Key Design: Để áp dụng thuật toán Sliding Window Counter mà ta đã thống nhất ở vòng trước, mỗi API Key (User) sẽ cần 2 key đại diện cho 2 cửa sổ phút (phút hiện tại và phút trước đó).

Định dạng: ratelimit:{apiKey}:{window_timestamp}

Ví dụ: Tại thời điểm phút thứ 29624580 (Epoch minutes), key sẽ là ratelimit:userX:29624580.

Atomic Increment (INCR / INCRBY): Redis là hệ thống đơn luồng (Single-threaded) xử lý các lệnh một cách tuần tự. Khi Server A, B, C đồng thời gọi lệnh INCR key, Redis sẽ xếp hàng và tăng giá trị bộ đếm lên +1 một cách tuyệt đối an toàn, loại bỏ hoàn toàn hiện tượng tranh chấp dữ liệu (Race Condition).

Expire Time & TTL (Time-To-Live): Ngay khi tạo key mới bằng lệnh tăng bộ đếm, Spring Boot Gateway phải gửi kèm lệnh EXPIRE key 120 (đặt TTL là 120 giây). Sau 2 phút, cửa sổ thời gian đó đã trôi qua quá khứ, Redis sẽ tự động xóa key này để giải phóng RAM, đảm bảo bộ nhớ không bị phình to theo thời gian.

3. Phân tích Ưu và Nhược điểm của Redis làm Rate Limiter
Ưu điểm:
Tốc độ đọc/ghi cực nhanh: Redis hoạt động hoàn toàn trên RAM, thời gian phản hồi cho mỗi lệnh đếm thường <2 ms, hầu như không làm tăng độ trễ (Latency) của SafePay API Gateway.

Chính xác tuyệt đối: Đảm bảo toàn bộ 3 cụm server Spring Boot đều nhìn chung một con số đếm, chặn đứng hành vi gửi lén lậu request.

Hỗ trợ cấu trúc dữ liệu tốt: Các lệnh nguyên tố cao giúp triển khai các biến thể Rate Limit phức tạp một cách dễ dàng.

Nhược điểm:
Thêm một điểm có thể gây lỗi (Infrastructure Overhead): Hệ thống phát sinh thêm một thành phần phụ thuộc (Dependency). Nếu Redis sập, toàn bộ logic kiểm tra tài khoản có thể bị tê liệt.

Chi phí hạ tầng nâng cao: Đòi hỏi kiến thức vận hành, giám sát tài nguyên RAM của Redis chặt chẽ.

4. What-if Scenarios (Kịch bản giả định & Giải pháp High Availability)
Nếu Redis bị chậm (High Latency)?

Cơ chế xử lý: Cài đặt Timeout ngắn (ví dụ: 15-20ms) cho kết nối từ Spring Boot đến Redis. Nếu quá thời gian này mà Redis chưa phản hồi, hệ thống lập tức kích hoạt cơ chế Fail-Open (Bỏ qua giới hạn - Cho phép request đi qua) để ưu tiên trải nghiệm người dùng thanh toán không bị gián đoạn, thay vì bắt người dùng đợi Redis.

Nếu Redis bị restart?

Cơ chế xử lý: Kích hoạt tính năng AOF (Append Only File) kết hợp RDB (Snapshotting) của Redis để ghi lại dữ liệu xuống đĩa cứng. Khi restart, Redis tự phục hồi lại bộ đếm. (Tuy nhiên, vì dữ liệu rate limit mang tính tạm thời trong 1 phút, việc mất mát vài giây dữ liệu trong lúc restart thường có thể châm chước được).

Nếu Redis Cluster mất 1 node?

Cơ chế xử lý: Sử dụng mô hình Redis Sentinel hoặc Redis Cluster với cơ chế Master-Slave. Khi một Node Master bị chết, các Node còn lại sẽ tự động bầu chọn (Failover) một Node Slave lên làm Master mới trong vòng vài giây mà Spring Boot không cần phải cấu hình lại IP.

Nếu Redis bị quá tải bộ nhớ (OOM / Out of Memory)?

Cơ chế xử lý: 1. Cấu hình chính sách giải phóng bộ nhớ: maxmemory-policy volatile-lru hoặc allkeys-lru. Khi RAM đầy, Redis tự xóa các key cũ nhất hoặc các key sắp hết hạn trước.
2. Triển khai cơ chế Local Cache Backup (Hybrid Rate Limiting): Nếu Spring Boot phát hiện kết nối Redis lỗi/quá tải, nó tự động hạ cấp (fallback) về sử dụng tạm bộ đếm local trong RAM trong vòng 5 phút để tự bảo vệ, sau đó thử kết nối lại Redis.

Quy trình Phân tích Kiến trúc (Chain-of-Thought)
Bước 1: Phân tích luồng xử lý request
Intercepting: Mỗi request đi qua SafePay API Gateway sẽ bị chặn lại ở tầng Filter/Interceptor trước khi chạm vào các Controller xử lý thanh toán.

Identification: Trích xuất X-API-KEY từ HTTP Header. Nếu không tìm thấy, từ chối ngay lập tức (HTTP 401).

Evaluation: Gửi API Key xuống RateLimiterService để kiểm tra trên Redis.

Decision: * Nếu hợp lệ (Count ≤ 100): Cho phép request đi tiếp (filterChain.doFilter()).

Nếu vượt hạn mức (Count > 100): Ném ra ngoại lệ, tầng Handler sẽ bắt lại và trả về HTTP 429 Too Many Requests.

Bước 2: Thiết kế Redis Key
Để tối ưu hóa thuật toán Sliding Window Counter xấp xỉ hoặc Fixed Window an toàn, ta thiết kế cấu trúc Key đính kèm mốc thời gian theo phút (Epoch Minutes):

Cấu trúc: safepay:ratelimit:{apiKey}:{epoch_minute}

Ý nghĩa: Việc đưa {epoch_minute} vào Key giúp phân tách rõ ràng bộ đếm của từng phút. Hệ thống không cần thực hiện lệnh xóa thủ công mà giao phó hoàn toàn cho cơ chế giải phóng bộ nhớ tự động của Redis thông qua TTL.

Bước 3: Mô tả logic nguyên tử (Atomic Logic)
Để tránh lỗi tranh chấp dữ liệu (Race Condition) trong hệ thống phân tán, ta sử dụng tính chất nguyên tử của lệnh INCR và EXPIRE phối hợp:

Khi request tới, thực hiện lệnh INCR(key).

Redis tăng giá trị và trả về con số hiện tại (currentCount).

Nếu currentCount == 1: Đây là request đầu tiên của cửa sổ phút này → Thực hiện ngay lệnh EXPIRE(key, 60) để thiết lập vòng đời cho key.

Nếu currentCount > 100: Ném ra RateLimitExceededException.

Bước 4: Phân tích cơ chế xử lý lỗi (Resilience - Khi Redis Sập)
Phương án A (Fail Open): Redis sập → Cho qua tất cả. Ưu điểm: Khách hàng không bị gián đoạn thanh toán. Nhược điểm: Hệ thống Core phía sau có nguy cơ bị sập nếu gặp DDoS.

Phương án B (Fail Closed): Redis sập → Chặn đứng toàn bộ. Ưu điểm: Bảo vệ Core tuyệt đối. Nhược điểm: Tự tay giết chết doanh thu cổng thanh toán.

Phương án C (Local Memory Fallback - Lựa chọn tối ưu cho SafePay): Khi phát hiện lỗi kết nối Redis (như QueryTimeoutException, RedisConnectionFailureException), hệ thống ghi nhận Log ERROR cảnh báo, đồng thời tự động hạ cấp xuống sử dụng một bộ đếm nội bộ trong RAM cục bộ (Caffeine Cache hoặc ConcurrentHashMap nội bộ) của chính instance Spring Boot đó như một lá chắn dự phòng trong vòng 1-5 phút cho đến khi Redis hồi phục.

Quyết định cho SafePay: Áp dụng Phương án C kết hợp Fail Open từng phần. Nếu Local RAM cũng quá tải, ta chấp nhận mở cửa (Fail Open) để bảo toàn luồng tiền thanh toán của doanh nghiệp, vì đối với Fintech, gián đoạn giao dịch (Downtime) mang lại thiệt hại nghiêm trọng hơn rủi ro tăng tải trong thời gian ngắn.

Bước 5: Thiết kế Hệ thống Logging Chuyên nghiệp
Sử dụng cấu trúc log có ngữ cảnh rõ ràng giúp các hệ thống như ELK, Splunk dễ dàng phân tích:

package com.safepay.gateway.config;

import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;
import org.springframework.data.redis.connection.RedisConnectionFactory;
import org.springframework.data.redis.core.RedisTemplate;
import org.springframework.data.redis.serializer.GenericToStringSerializer;
import org.springframework.data.redis.serializer.StringRedisSerializer;

@Configuration
public class RedisConfig {

    @Bean
    public RedisTemplate<String, Long> redisTemplate(RedisConnectionFactory connectionFactory) {
        RedisTemplate<String, Long> template = new RedisTemplate<>();
        template.setConnectionFactory(connectionFactory);
        
        // Sử dụng StringSerializer cho Key và Long GenericToStringSerializer cho Value bộ đếm
        template.setKeySerializer(new StringRedisSerializer());
        template.setValueSerializer(new GenericToStringSerializer<>(Long.class));
        template.setHashKeySerializer(new StringRedisSerializer());
        template.setHashValueSerializer(new GenericToStringSerializer<>(Long.class));
        
        template.afterPropertiesSet();
        return template;
    }
}

3. Thành phần Cốt lõi: RateLimiterService với Local Fallback
Java
package com.safepay.gateway.service;

import com.safepay.gateway.exception.RateLimitExceededException;
import lombok.RequiredArgsConstructor;
import lombok.extern.slf4j.Slf4j;
import org.springframework.data.redis.core.RedisTemplate;
import org.springframework.stereotype.Service;

import java.time.Duration;
import java.time.Instant;
import java.util.concurrent.ConcurrentHashMap;
import java.util.concurrent.atomic.AtomicLong;

@Slf4j
@Service
@RequiredArgsConstructor
public class RateLimiterService {

    private final RedisTemplate<String, Long> redisTemplate;
    private static final long LIMIT = 100L;
    
    // Bộ lưu trữ dự phòng cục bộ phòng khi Redis sập (Local Memory Fallback)
    private final ConcurrentHashMap<String, AtomicLong> localFallbackCache = new ConcurrentHashMap<>();

    public void checkRateLimit(String apiKey) {
        long currentEpochMinute = Instant.now().getEpochSecond() / 60;
        String redisKey = String.format("safepay:ratelimit:%s:%d", apiKey, currentEpochMinute);

        try {
            // Lệnh nguyên tử tăng giá trị đếm trên Redis
            Long currentCount = redisTemplate.opsForValue().increment(redisKey);

            if (currentCount == null) {
                return; // Phòng vệ rủi ro hiếm gặp từ Redis driver
            }

            // Nếu là request đầu tiên trong phút này, gán TTL tự hủy sau 60s
            if (currentCount == 1) {
                redisTemplate.expire(redisKey, Duration.ofSeconds(60));
            }

            if (currentCount > LIMIT) {
                log.warn("[RATE_LIMIT][BLOCKED] Key: {}, CurrentCount: {}, Limit: {}", apiKey, currentCount, LIMIT);
                throw new RateLimitExceededException("API rate limit exceeded. Maximum allowed is 100 requests per minute.");
            }

            log.info("[RATE_LIMIT][ACCEPTED] Key: {}, CurrentCount: {}", apiKey, currentCount);

        } catch (RateLimitExceededException e) {
            throw e; // Ném tiếp ra ngoài nếu là lỗi vượt hạn mức thực tế
        } catch (Exception redisException) {
            // Kích hoạt cơ chế hạ cấp an toàn Local Memory Fallback khi Redis lỗi
            log.error("[RATE_LIMIT][FALLBACK_ACTIVATED] Redis unavailable. Error: {}. Switching to local counter.", redisException.getMessage());
            handleLocalFallback(apiKey, currentEpochMinute);
        }
    }

    private void handleLocalFallback(String apiKey, long epochMinute) {
        String localKey = String.format("%s:%d", apiKey, epochMinute);
        
        // Tự động giải phóng map local nếu đổi sang phút mới nhằm tránh rò rỉ bộ nhớ RAM
        if (localFallbackCache.size() > 5000) { 
            localFallbackCache.clear(); 
        }

        AtomicLong counter = localFallbackCache.computeIfAbsent(localKey, k -> new AtomicLong(0));
        long currentLocalCount = counter.incrementAndGet();

        if (currentLocalCount > LIMIT) {
            log.warn("[RATE_LIMIT][LOCAL_BLOCKED] Key: {}, LocalCount: {}", apiKey, currentLocalCount);
            throw new RateLimitExceededException("API rate limit exceeded (Local Fallback Protection).");
        }
        
        log.info("[RATE_LIMIT][LOCAL_ACCEPTED] Key: {}, LocalCount: {}", apiKey, currentLocalCount);
    }
}
4. Tầng chặn lọc Request: RateLimitingFilter
Java
package com.safepay.gateway.filter;

import com.safepay.gateway.exception.RateLimitExceededException;
import com.safepay.gateway.service.RateLimiterService;
import jakarta.servlet.*;
import jakarta.servlet.http.HttpServletRequest;
import jakarta.servlet.http.HttpServletResponse;
import lombok.RequiredArgsConstructor;
import org.springframework.core.annotation.Order;
import org.springframework.stereotype.Component;

import java.io.IOException;

@Component
@Order(1) // Đảm bảo Filter chạy đầu tiên trong ứng dụng
@RequiredArgsConstructor
public class RateLimitingFilter implements Filter {

    private final RateLimiterService rateLimiterService;

    @Override
    public void doFilter(ServletRequest request, ServletResponse response, FilterChain chain)
            throws IOException, ServletException {
        
        HttpServletRequest httpRequest = (HttpServletRequest) request;
        HttpServletResponse httpResponse = (HttpServletResponse) response;

        String apiKey = httpRequest.getHeader("X-API-KEY");

        if (apiKey == null || apiKey.trim().isEmpty()) {
            httpResponse.setStatus(HttpServletResponse.SC_UNAUTHORIZED);
            httpResponse.getWriter().write("Missing API Key (Header: X-API-KEY)");
            return;
        }

        try {
            rateLimiterService.checkRateLimit(apiKey);
            chain.doFilter(request, response);
        } catch (RateLimitExceededException ex) {
            httpResponse.setStatus(HttpServletResponse.SC_TOO_MANY_REQUESTS); // HTTP 429
            httpResponse.setContentType("application/json");
            httpResponse.getWriter().write(String.format("{\"error\": \"Too Many Requests\", \"message\": \"%s\"}", ex.getMessage()));
        }
    }
}
